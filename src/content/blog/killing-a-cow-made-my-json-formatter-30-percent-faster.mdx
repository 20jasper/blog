A formatter styles code consistently, generally improving human readability. If you are unfamiliar with formatters, tokenizers, or abstract syntax trees, see my article on [how formatters work](TODO)

The JavaScript ecosystem has a gap—most formatters either are featureful _or_ fast. Prettier is the standard in the JavaScript ecosystem, but it's not the fastest. Oxfmt has made major strides—supporting parity with Prettier's JavaScript formatting with much improved performance. [One report found a migration to Oxfmt formatted their repo 6.5x faster—from 13.9 seconds to 2.1](https://www.udaynayak.com/blog/migrating-from-eslint-prettier-to-oxlint-oxfmt-in-an-nx-monorepo)

However these gains do not apply to JSON. Oxfmt is 10–20% slower than Prettier for JSON. Oxfmt has a native JavaScript formatter, but it reasonably delegates to Prettier for file types without native implementations[^respectOxfmt]

[^respectOxfmt]: I love everything Oxfmt has done so far, so this is not meant to put down Oxfmt. It's the only Prettier compatible tooling I've found that doesn't crash when formatting larger JSON files. I've seen some amazing architectural decisions they've made. Open source is a lot of work

Prettier and Oxfmt both take more than a second to format a 2MB JSON file, while my formatter, [JJPWRGEM](TODO) takes around 30 milliseconds. These comparisons are unfair right now since JJPWRGEM is in the "fast but not so many features" camp right now

## Number normalization

Prettier does several numeric transformations, but I'll start with these 3 related to scientific notation for simplicity

1. Lowercases exponent symbol (from `1E5` to `1e5`)
2. Strips `+` (from `1e+5` to `1e5`)
3. Removes 0 exponents (from `1e0` to `1`)

### Baseline

At the moment, JJPWRGEM leaves numbers untouched and represents them as a single `Cow<str>`

I am benchmarking against several files with varying data shapes

Deser refers to deserializing into an abstract syntax tree. The performance is represented by throughput to avoid size based discrepancies. If we have a 1MB file and a 2MB file, the smaller will be formatted faster, but `MB/s` is comparable for both!

| benchmark                       | baseline (MB/s) |
| ------------------------------- | --------------- |
| deser/dense floats              | 115.6           |
| deser/many objects and integers | 385.5           |
| deser/strings and deep nesting  | 257.8           |

### Iteration 0: Build a new normalized string

The simplest approach is to build new strings for improperly formatted numbers

Unfortunately with this approach, each number needs traversed twice, once to find the range and again to validate it. Even if there are no numbers in our input, this ties us to the string cleanup machinery taking 20% of our instructions!

### Iteration 1: 2 `Cow`s

We only care about 2 portions of the string for each number, the part to the left of the `e`, or the mantissa, and the part to the right--the exponent. You may recognize this as scientific notation—`1e5` represents $1e5$

Say we have `1e+05`, we can store `1` and `5` and reconstruct `1e5`. If we have an exponent, the `e` must be lowercase and any `+`s can be skipped

```rust
pub enum Token<'a> {
    Number {
        mantissa: Cow<'a, str>,
        exponent: Cow<'a, str>
    },
}
```

This works, but lowers performance across the board, even for datasets without many numbers. This is because I doubled our unnecessary `Cow` cleanup. Any `Token` could be a `Number`—the compiler has to prepare for potential `Cow` cleanup even if it may never come!

| benchmark                       | baseline (MB/s) | this (MB/s) | delta  |
| ------------------------------- | --------------- | ----------- | ------ |
| deser/dense floats              | 115.6           | 101.1       | -12.5% |
| deser/many objects and integers | 385.5           | 289.3       | -25.0% |
| deser/strings and deep nesting  | 257.8           | 214.8       | -16.7% |

Let's start with getting rid of the `Cow`s!

### Iteration 2: use `&str` instead of `Cow<str>`

```rust
pub enum Token<'a> {
    Number {
        mantissa: &'a str,
        exponent: &'a str
    },
}
```

The compiler gets to remove all those cleanup instructions and branches, which is a huge win for most datasets

However, dense floats is below baseline since it has no exponents—it's paying the cost to check for exponents that never materialize. Let's try splitting the tokens in 2 to avoid paying that cost

| benchmark                       | baseline (MB/s) | this (MB/s) | delta  |
| ------------------------------- | --------------- | ----------- | ------ |
| deser/dense floats              | 115.6           | 111.0       | -3.9%  |
| deser/many objects and integers | 385.6           | 442.9       | +14.9% |
| deser/strings and deep nesting  | 258.7           | 341.4       | +32.0% |

### Iteration 3: Separate `Mantissa` and `Exponent` variants

Instead of 1 variant holding the full number, we'll split it in 2 so the CPU can read 2 tokens at once, saving an instruction

```rust
pub enum Token<'a> {
    Mantissa(&'a str),
    Exponent(&'a str),
}
```

CPUs read memory in chunks—like reading a word at a time instead of a full sentence

| ------------------------------- | ----------- | ----------- | ------ |
| deser/dense floats              | 111.0       | 122.6       | +10.5% |
| deser/many objects and integers | 442.9       | 401.7       | -9.3%  |
| deser/strings and deep nesting  | 341.4       | 356.8       | +4.5%  |

### Iteration 4: Split the state machine

Parsing both the mantissa and exponent has ballooned number state machine size to 88 bytes, meaning the CPU needs two reads

A state machine is like a sequential checklist. To find where a number starts and ends, we start with a digit or minus sign, then another digit or a `.` for a float. At first, it made sense to have one state machine since we only cared about one chunk. Now we care about mantissa and exponent

Not only will splitting allow the CPU to fetch the full state in one read, the compiler has an easier time optimizing fewer states. It can make better decisions with less possibilities to reason about

```rust
enum MantissaState { ... }
enum ExponentState  {
    MinusOrPlusOrDigit,
    AfterSign,
    Digits,
    Zero,
    End
}
```

Now the regression in the many objects and integers dataset is fixed and even faster than the baseline! Unfortunately, dense floats has regressed to slightly below baseline, but that's an acceptable tradeoff since typical JSON has more mixed data

| benchmark                       | prev (MB/s) | this (MB/s) | delta  |
| ------------------------------- | ----------- | ----------- | ------ |
| deser/dense floats              | 122.6       | 111.3       | -9.2%  |
| deser/many objects and integers | 401.7       | 456.9       | +13.7% |
| deser/strings and deep nesting  | 356.8       | 385.1       | +7.9%  |

## Final Benchmarks

Switching from `Cow` to `str` and splitting `Number` token and state machines both help to improve performance

### Deserialization into AST

From string to AST. dense floats pays for the extra unused token per number, but many objects and integers and strings with deep nesting have massive improvements due to converting `Cow` to `str`

|                           | baseline (MB/s) | new (MB/s) | delta  |
| ------------------------- | --------------- | ---------- | ------ |
| dense floats              | 115.6           | 111.3      | -3.7%  |
| many objects and integers | 385.5           | 456.9      | +18.5% |
| strings and deep nesting  | 257.8           | 385.1      | +49.4% |

### Prettify from AST

From AST to human readable string. dense floats suffered the most for a branch on each `Number` to check if an exponent is available to print. Other datasets are flat

|                           | baseline (MB/s) | new (MB/s) | delta |
| ------------------------- | --------------- | ---------- | ----- |
| dense floats              | 1424.7          | 1332.0     | -6.5% |
| many objects and integers | 1853.2          | 1845.3     | -0.4% |
| strings and deep nesting  | 2091.1          | 2105.0     | +0.7% |

### Uglify from Token Stream

Going from string to tokens to string. Big improvements across the board. Avoids building a syntax tree, so serialization avoids branching with formatting like with AST to string paths

|                           | baseline (MB/s) | new (MB/s) | delta  |
| ------------------------- | --------------- | ---------- | ------ |
| dense floats              | 198.9           | 251.2      | +26.3% |
| many objects and integers | 449.8           | 630.4      | +40.2% |
| strings and deep nesting  | 269.9           | 418.2      | +54.9% |

## What's a `Cow` anyways?

`Cow` stands for Clone On Write. It references data until mutation is needed, then duplicates it. This pays off when most values are left unchanged—only the data that needs modifications is cloned

A `to_lowercase` function is a great use case—if the input is already lowercase, you can return a reference to the same string! In Rust terms, the data is "borrowed"

```rust
fn to_lowercase(string: &str) -> Cow<'_, str> {
    if string.is_lowercase() {
        // reference input string
        Cow::Borrowed(string)
    } else {
        // make a new string
        Cow::Owned(string.to_lowercase())
    }
}
```

<FigCaption src="/cow/lowercase-cow.webp">
	Diagram showing a call to `to_lowercase` with an already-lowercase `&str`
	returns a `Cow::Borrowed` pointing to the same memory, avoiding allocation
</FigCaption>

Now imagine an uppercase input. A new allocation is made to avoid changing the referenced string. The `Cow` "owns" the new data and can modify within the allocated capacity without reallocating

<FigCaption src="/cow/uppercase-cow.webp">
	Diagram showing a call to `to_lowercase` with an uppercase `&str` returns a
	`Cow::Owned` pointing to a new allocation
</FigCaption>

Sounds great right? That's why I used a `Cow` to represent numbers

## Premature optimization is bad

While `Cow` gave me flexibility, I never took advantage, hurting performance without any gain

Since a `Cow` is potentially responsible to clean up its data, a small performance penalty is paid each time it goes out of scope, ultimately rendering 20% of the program's instructions useless!
