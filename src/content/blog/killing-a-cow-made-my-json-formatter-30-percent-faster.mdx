## What's a `Cow` anyways?

`Cow` stands for Clone On Write—referencing data until mutated, when it's duplicated. This is useful when most values will be untouched, but you might want to mutate some later

For a `to_lowercase` function—if the string is already lowercase, you can just return a reference to the same string! In Rust terms, the data is "borrowed"

```rust
fn to_lowercase(string: &str) -> Cow<'_, str> {
    if string.is_lowercase() {
        // reference input string
        Cow::Borrowed(string)
    } else {
        // make a new string
        Cow::Owned(string.to_lowercase())
    }
}
```

Now imagine an uppercase input. A new allocation is made to avoid changing the referenced string. The `Cow` "owns" the new data and can modify within the allocated capacity without reallocating

## premature optimization is bad

Around 20% of instructions were unnecessary due to using a `Cow<str>` instead of a `&str`

I jumped to flexibility too soon. I used a `Cow<str>` to represent numbers so that consumers of my API could pass either `"12"` or `20.to_string()`. Did I ever have a use case for that? Nope

Since a `Cow` can potentially own its data, it also needs to cleanup or "drop" it. This is a small amount of overhead, but it adds up when called in a hot loop

Even worse, JSON has more than just numbers. The formatter parses chunks of text into meaningful units called tokens, such as `OpenCurlyBrace` and `Null`. Since each token could be a `Number`, the compiler needs to check if the `Cow` needs to cleanup its data

Changing to the `Number` to hold a reference to a `str` removes around 20% of unnecessary instructions (around 12 million for a 1.7MB file). A followup deep dive into LLVM IR proving it's not memory layout or branch predictions is on the way!

## How does a formatter work?

Before I explain what I did, you need to understand how a formatter works

1. convert a string input into an in memory representation (deserialization)
2. output a string according to formatting rules (serialization)

For example, JJPWRGEM converts strings to meaningful chunks called tokens

{/* prettier-ignore */}
```json
{"hi hello"    : 10 }
```

```rust
[OpenCurlyBrace, String("hi hello"), Colon, Number("10"), ClosedCurlyBrace]
```

To uglify, or remove unnecessary characters, we can remove any insignificant whitespace like spaces outside of strings or newlines. It's as simple as appending each token's string representation to the output

{/* prettier-ignore */}
```json
{"hi hello":10}
```

For more complex cases like pretty printing, we can build an abstract syntax tree, or AST, with the tokens. An AST encodes relationships like nesting and key value pairs. Prettier and JJPWRGEM use an abstract syntax tree to quickly calculate how wide an object would be on a single line, and expand if it's too large

```rust
Object([
	Entry(String("hi hello"), Number("10"))
])
```

{/* prettier-ignore */}
```json
{
	"hi hello": 10
}
```

## Number normalization

Prettier does several numeric transformations, and we'll handle the following 3

1. Lowercases exponent symbol (from `1E5` to `1e5`)
2. Strips `+` (from `1e+5` to `1e5`)
3. Removes 0 exponents (from `1e0` to `1`)

So how do we go about implementing this without hurting performance? Let's walk through each approach I tried and how it affected performance

### Baseline

At the moment, JJPWRGEM leaves numbers untouched and represents then as a single `Cow<str>`

### Approach 1: Build a new normalized string

The simplest approach is to check if we need to build a new string and replace it if not. This is a great usecase for `Cow`s after all!

Unfortunately, each number needs traversed twice, once to find the range and again to validate it. Even if there are no numbers in our input, heap allocations are slow and this ties us to the string cleanup machinery taking 20% of our instructions!

### Approach 2: 2 `Cow`s

We only care about 2 portions of the string for each number, the part to the left of the `e`, or the mantissa, and the part to the right--the exponent

Say we have `1e+05`, we can store `1` and `5` and reconstruct `1e5`. If we have an exponent, the `e` must be lowercase and any `+`s can be skipped

```rust
pub enum Token<'a> {
    Number {
        mantissa: Cow<'a, str>,
        exponent: Cow<'a, str>
    },
}
```

This works, but lowers performance, which is measured by throughput to avoid size based discrepancies. If we have a 1MB file and a 2MB file, the smaller will be formatted faster, but `MB/s` is comparable for both! Deser refers to deserializing into an abstract syntax tree

### Approach 3: use `&str` instead of `Cow<str>`

```rust
pub enum Token<'a> {
    Number {
        mantissa: &'a str,
        exponent: &'a str
    },
}
```

The compiler gets to remove all those cleanup instructions and branches, which is a huge win!

dense floats is below baseline, which checks out since it has no exponents, so it's paying the cost to check for exponents every time.

| benchmark                       | baseline (MB/s) | this (MB/s) | delta  |
| ------------------------------- | --------------- | ----------- | ------ |
| deser/dense floats              | 115.6           | 111.0       | -3.9%  |
| deser/many objects and integers | 385.6           | 442.9       | +14.9% |
| deser/strings and deep nesting  | 258.7           | 341.4       | +32.0% |

### Approach 4: Separate `Mantissa` and `Exponent` variants

Instead of 1 variant holding both parts of the Number, we'll split them up so the CPU can read 2 tokens at once

```rust
pub enum Token<'a> {
    Mantissa(&'a str),
    Exponent(&'a str),
}
```

Most CPUs read 64 bytes at a time, so 2 `&str` or `Cow<str>` per token limited reads to one `Token` at a time

Variants holding 1 `&str` or `Cow<str>` are both 24 bytes, so the 2 fit in 1 read

### Approach 5: Split the state machine

The same principle applies to our Number state machine. This makes sure we only accept valid numbers. For example, numbers can only start with a minus sign or a non zero integer in JSON

Since we started outputting 2 separate `&str` and whether there's a negative exponent, our `NumberState` has ballooned to 88 bytes. It was reasonable to have 1 state machine with one output, but the CPU needs to read 2 times for each state change. Splitting them puts us back at the baseline of 1 read

```rust
enum MantissaState { MinusOrInteger, IntegerOrDecimalOrEnd, Fraction, FractionOrEnd, End }
enum ExponentState  { MinusOrPlusOrDigit, AfterSign, Digits, Zero, End }
```

Additionally, fewer states means fewer options, so the CPU can better predict what will happen and the compiler has an easier time optimizing

## Final Benchmarks

Switching from `Cow` to `str` and splitting `Number` token and state machines both help to improve performance

### Deserialization into AST

From string to AST. dense floats pays for the extra unused token per number, but many objects and integers and strings with deep nesting have massive improvements due to `Cow` to `str`

|                           | baseline (MB/s) | new (MB/s) | delta  |
| ------------------------- | --------------- | ---------- | ------ |
| dense floats              | 115.6           | 111.3      | -3.7%  |
| many objects and integers | 385.5           | 456.9      | +18.5% |
| strings and deep nesting  | 257.8           | 385.1      | +49.4% |

### Prettify from AST

From AST to human readable string. dense floats suffered the most for a branch on each `Number`. Other datasets are flat

|                           | baseline (MB/s) | new (MB/s) | delta |
| ------------------------- | --------------- | ---------- | ----- |
| dense floats              | 1424.7          | 1332.0     | -6.5% |
| many objects and integers | 1853.2          | 1845.3     | -0.4% |
| strings and deep nesting  | 2091.1          | 2105.0     | +0.7% |

### Uglify from Token Stream

Going from string to tokens to string. Big improvements across the board. Avoids building a syntax tree, so serialization avoids branching with formatting like AST uglification

|                           | baseline (MB/s) | new (MB/s) | delta  |
| ------------------------- | --------------- | ---------- | ------ |
| dense floats              | 198.9           | 251.2      | +26.3% |
| many objects and integers | 449.8           | 630.4      | +40.2% |
| strings and deep nesting  | 269.9           | 418.2      | +54.9% |
